# Impeccable — project design context

_Last updated: teach-impeccable pass (codebase scan). Refine after stakeholder answers to the open questions below._

## Design Context

### Users

**Inferred from the codebase:** Primary users are **operators and builders** running HSM-II locally or in their stack: people who configure agents, Company OS workspaces, eval harnesses, and the **company console** (Next.js). They work in **technical context**—terminals, Postgres, `hsm_console`, API proxies—and need **dense, scannable** UIs for tasks, governance, spend, marketplace packs, and playbooks.

**Jobs to be done (inferred):** Monitor agent/workflow health, triage human-in-the-loop queues, install Paperclip-style packs, author SOPs, inspect graphs/trails/memory, and run intelligence-layer goals without losing state across sessions.

**Still to confirm with you:** Whether **non-technical stakeholders** (execs, ICs in other departments) are in scope for the console, or if this stays **engineer/operator-first** only.

### Brand Personality

**Inferred:** **Precise, industrial, calm under load** — aligned with an in-repo Nothing-inspired system: monochrome OLED base, **structure as ornament**, monospace for system labels, restrained color for **signal** (destructive red `#d71921`, primary blue `#5b9bf6`, success/warn tokens). Secondary personality from **Paperclip-class** admin: GitHub-dark panels (`admin.*`), dense tables, instrument-style hierarchy.

**Three-word handle (proposed):** _Technical, stigmergic, legible._

**Emotional target (inferred):** **Confidence and control** (dashboards, live counts, clear borders) more than playfulness; **urgency** only where encoded (destructive, warnings).

**Still to confirm:** Your own **three words**, any **reference products** (beyond Nothing/Paperclip/GitHub-dark already echoed in code), and **anti-references** (“must not feel like X”).

### Aesthetic Direction

**From code:** **Dark-first** (`globals.css`: OLED black background, raised `#111` surfaces). **One ambient layer** on dashboard shell (`nd-dashboard-shell` radial gradient); **cards stay flat** per comments. Typography: **Space Grotesk** + **Space Mono** (tailwind). **ALL CAPS** micro-labels (`nd-label`, `pc-page-eyebrow`, table headers). **shadcn/Radix** primitives mapped to the same tokens.

**Light mode:** Tokens exist in nothing-design skill for dual mode, but **this app’s `:root` is explicitly dark**; treat **light as secondary** unless product decides otherwise.

**Still to confirm:** Whether you need **first-class light mode** for the company console soon, and any **mandatory or forbidden brand colors** beyond current tokens.

### Design Principles

1. **Subtract before decorate** — Every new chrome must earn its pixels; prefer spacing and type hierarchy over extra boxes and icons (Nothing skill alignment; already reflected in `pc-*` utilities and flat panels).

2. **Color signals state, not decoration** — Reserve red/blue/green/amber for status, links, and danger; keep surfaces near black/gray unless encoding data (matches `--destructive`, `--primary`, chart semantics).

3. **Monospace = system truth** — IDs, API hints, table headers, and labels use mono + tracking; prose stays in sans for readability (`tailwind` font stacks).

4. **One calm ambient, flat content** — Dashboard may use a single background treatment; widgets remain **flat bordered panels** so data stays scannable (`globals.css` comment pattern).

5. **Density is a feature, not a bug** — Tables, sidebars, and multi-panel layouts are intentional for **power users**; simplify only when a surface is explicitly “consumer” scope.

6. **Explicit Nothing-design skill** — For work tagged Nothing style, `/nothing-design`, or gradients (`nothing-gradients` skill), follow those skills **on top of** this context—they are stricter for typography and motion.
